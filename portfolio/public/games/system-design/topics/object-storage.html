Object Storage 物件存儲系統

我教你設計大檔案上傳同 CDN 分發，好似 S3 咁嘅架構

我帶你理解 Object Storage

唔係 file system，係 key-value 存大檔案

我要同你講清楚：傳統檔案系統係 hierarchy（folder/file），但 Object Storage（例如 AWS S3）用 bucket + key 嘅方式存儲。每個 object 有唯一 key，value 就係檔案內容（blob）。你要記住一個重點：metadata（檔名、size、content-type）同 actual data 係分開存嘅，各司其職。

我建議你一定要掌握兩個關鍵設計：大檔案上傳要 support Multipart Upload；分發要用 CDN 加速下載。呢兩樣嘢係 Object Storage 嘅核心。

我教你成個流程

  1. 1Client 想上傳大檔案，先問 Upload Service 攞 Presigned URL。我要同你強調：呢步好關鍵，因為之後 Client 可以直接 upload 去 S3，唔使經你嘅 Server。
  2. 2Client 直接 upload 去 S3，大幅減輕 backend 負擔。你要記住，呢個就係 Presigned URL 嘅威力。
  3. 3Metadata DB 記錄 object key、path、size、上傳時間等。我建議你將 metadata 同 blob 分開，方便搜尋。
  4. 4下載時經 CDN，熱門檔案喺 edge 快取。我想你記住：全球用戶就近下載，速度快好多。

我教你上傳策略

Presigned URL 同 Multipart Upload

我要同你講兩個最重要嘅上傳機制。Presigned URL：Server 生成一個帶簽名嘅 URL，Client 可以直接用呢個 URL 上傳或下載，唔使經過你嘅 API。限時有效，安全又減負載。你一定要掌握呢個設計。

Multipart Upload：大檔案（例如 >100MB）分做多個 part 上傳，最後 merge。我想你記住佢嘅優勢：支援斷點續傳，失敗咗只需重傳失敗嘅 part，唔使由頭嚟。

Presigned URL

Server 簽名 URL，Client 直接同 S3 溝通。限時（例如 15 min），過期無效。我建議你用呢個方式，唔使暴露 AWS key。

Multipart Upload

分 part 上傳（每 part 5MB–5GB），流程係 Initiate → Upload parts → Complete。你可以並行上傳多個 part，大幅加速大檔案傳輸。

CDN 分發

熱門檔案 cache 喺 edge 節點，用戶就近下載。我想你記住：CDN 減少 origin 壓力，加快全球訪問速度。

Metadata 同 Blob 分開

Metadata（檔名、大小、path）放 DB 方便搜尋；blob 放 Object Store 低成本存儲。我建議你一定要分開，各司其職先係正確設計。

實戰要點

我幫你整理設計 Object Storage 要諗嘅嘢

Part Size

Multipart 最小 5MB（最後 part 可以細啲）。我建議你用 5-10MB 做 balance：太大 part 失敗重傳成本高；太細 overhead 多。

斷點續傳

記錄已上傳嘅 part，resume 時跳過。你要 persist upload_id 同 completed parts，呢步唔好偷懶。

Abort 清理

用戶 cancel 或者超時，你一定要 call Abort Multipart Upload。我提醒你：orphan parts 會佔空間收錢，唔清理會浪費資源。

權限控制

Presigned URL 可以限制 HTTP method、content-type、expiry。我建議你下載都用 signed URL + CDN，安全又快。

我嘅總結

AWS S3、Google Cloud Storage、阿里雲 OSS 都係 Object Storage。好多大型平台背後都用類似架構：上傳經 Presigned/Multipart，下載經 CDN。你掌握咗呢套設計，處理任何大檔案存儲場景都有信心。
