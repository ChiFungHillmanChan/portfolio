點解寫 Code 從來都唔係工程師嘅核心工作

AI 取代工程師？— 正反雙方

留意：叫人轉行嘅人，自己幾乎從來都唔係真正嘅工程師

一個好大膽嘅問題：「喺 AI 年代，Software Developer 係咪應該轉行？」有啲人會話「絕對係」，但事實並非如此。留意下，叫人轉行嘅人，自己幾乎都唔係做 Software Engineering 嘅。必須學識分辨呢啲聲音。

「叫人轉行嘅人，自己幾乎從來都唔係真正做 Software Engineering 嘅。」

核心論點

  1. 1 管理層同媒體嘅觀感同前線工程師嘅實際體驗有好大落差——Leaders 覺得 AI 好勁，但大部分 Individual Contributors 話提速好有限。唔好被表面嘅聲音嚇到。
  2. 2 話工程師會被取代嘅人，幾乎都唔係自己做 Engineering 嘅。呢啲人可能低估咗工程師真正做緊嘅嘢。必須識分辨。
  3. 3 呢個落差嘅根本原因：大部分人從來冇真正理解過 Software Engineer 實際做緊乜嘢。以下會詳細解釋。

值得反思

如果聽到有人話「AI 會取代工程師」，先問下：講呢句嘢嘅人，有冇實際做過 Engineering？有冇理解工程師日常做緊啲咩？養成呢個批判思維好重要。

寫 Code 從來都唔係主要工作

重點：Writing code has never been the main part of the job

好多人以為 Software Engineer = 寫 Code 嘅人。但事實係：寫 Code 從來都唔係主要嘅工作。大部分時間同精力，其實係花喺設計系統、思考 Trade-off、確保系統可靠同可擴展。以下講解點樣正確理解工程師嘅角色。

「Software Engineer 嘅使命係解決已知嘅問題，同時發現新嘅問題去解決。寫 Code 只係其中一個任務。」

點解大家會誤解

  1. 1 外行人見到工程師日日對住 IDE 打字，就以為工作就係打字。呢個好似見到醫生揸筆寫嘢，就以為工作係寫字咁荒謬。
  2. 2 寫 Code 只係將「已經想好嘅方案」實現出嚟。真正花時間嘅係——想個方案出嚟。
  3. 3 AI 可以幫手快啲寫完 Code，但幫唔到決定「應該點樣設計個系統」、「呢兩個方案邊個好」。必須記住呢個分別。

AI 幫到手嘅地方

  • 自動補全 Code
  • 寫 Boilerplate / 重複性嘅 Code
  • 快速查文件同 API 用法
  • 寫測試同 Debug 輔助

AI 暫時幫唔到嘅地方

  • 判斷邊個架構方案最適合
  • 衡量 Downtime 風險同 Trade-off
  • 同 Stakeholder 溝通需求同優先級
  • 處理跨團隊嘅系統依賴關係

值得反思

如果覺得自己份工係「寫 Code」，咁真係有機會被 AI 取代。但如果理解自己嘅角色係「解決問題」，AI 只係手上多咗一件好好用嘅工具。呢個就係關鍵嘅思維轉變。

市場需求反而增加

每一間公司而家都想 build 更多 software，唔係更少

好多人驚 AI 會令工程師冇工做。但現實恰恰相反——每一間公司而家都想 build 更多 software，唔係更少。AI 唔係令工程師消失，而係令工程師更加有價值，因為公司需要工程師去善用 AI 嘅能力嚟 build 得更快。以下詳細解釋。

點解需求反而增加

  1. 1 Startup 競爭加劇：個個都用 AI 快速 build 產品，大公司要跟上就要 build 更多、更快。必須明白呢個連鎖反應。
  2. 2 大公司之間嘅軍備賽：競爭對手用 AI 推快咗開發速度，唔跟就落後。呢個趨勢只會加速。
  3. 3 工程師有 Leverage：公司視工程師為有能力善用 AI 去加速 build 嘅關鍵人物。呢個價值必須認識到。
  4. 4 Downtime 成本極高：Scale 大嘅公司，1 小時故障可以蝕百萬到幾億美金，遠超請工程師嘅成本。必須理解呢個經濟邏輯。

對工程師有利嘅趨勢

  • 公司想 build 更多 software，唔係更少
  • AI 令工程師嘅生產力提升 = 更有價值
  • 系統越嚟越複雜，更加需要專業判斷
  • 懂用 AI 嘅工程師會更搶手

需要留意嘅風險

  • 只識寫 Code 但唔識思考嘅人，真係會被淘汰
  • 唔肯學用 AI 工具嘅人會被拋離
  • Junior 工程師嘅入行門檻可能會變高
  • 某啲重複性高嘅工作確實會被 AI 取代

結論

AI 唔係敵人，而係工具。真正有風險嘅唔係「工程師呢個職業」，而係「唔肯進步嘅工程師」。重點就係點樣做一個會進步嘅工程師。

重點總結 — 並冇被 Cook

結論：工程師嘅工作從來都唔係寫 Code，而係解決問題

結論好清晰：如果係 Software Engineer 或者打算入行，並唔係「被 cook 咗」。只需要認清一件事——工程師嘅工作從來都唔係寫 Code，而係解決問題。帶住呢個認知繼續行落去。

「如果係 Software Engineer 或者打算入行，並冇被 cook。只需要認清工程師嘅工作從來都唔係寫 Code。」

最後忠告

  1. 1 並冇被「Cook」。Software Engineering 唔會消失，但定義會進化。做一個適應能力強嘅工程師——呢種人會更加吃香。
  2. 2 重新理解工程師嘅價值。價值唔係喺打字速度，而係喺思考能力——設計系統、權衡利弊、確保穩定。呢啲就係核心。
  3. 3 擁抱 AI，但唔好依賴。AI 幫手寫得更快，但邊度要寫、點解要寫、寫啲乜——呢啲判斷仲係工程師嘅。必須牢牢掌握呢個主導權。
  4. 4 由今日開始，學系統設計。呢個正正係本課程嘅目的——由「寫 Code 嘅人」進化做「設計系統嘅人」。跟住課程學落去就啱。

最後一句

揀咗入呢個教室，已經證明唔係淨係識寫 Code 嘅人。繼續學落去，價值只會越嚟越高。
