Message Queue 分佈式消息隊列

事件驅動架構（Kafka），pub/sub 模式

我教你 Message Queue 係咩

Producer 發消息，Consumer 訂閱消費，解耦系統組件

我要同你講一個好重要嘅概念。當你嘅系統有好多服務要互相溝通——Order Service 下單之後，要通知 Inventory Service 扣庫存、Payment Service 收錢、Notification Service 發通知——如果每個都直接 call 對方嘅 API，你知會點？會變成蜘蛛網，改一個牽動全身。我見過太多初學者犯呢個錯。

我建議你用 Message Queue（例如 Kafka）嚟解決呢個問題。做法好簡單：Order Service 發一條「訂單已建立」嘅消息去 Kafka，其他服務訂閱呢個 topic，自己 pull 消息嚟處理。呢個就係解耦！你要記住：加新服務只係加多個 consumer，完全唔使改 Order Service。

我話你知點樣運作

  1. 1Producer 發消息去 Kafka 嘅 Topic。你要知道，每個 Topic 會分拆成多個 Partition，呢個就係水平擴展嘅關鍵。
  2. 2Consumer Group 訂閱 Topic，我要你記住：每個 Partition 只會被 Group 入面一個 Consumer 處理。所以你加多幾台 Consumer 就可以並行消費。
  3. 3ZooKeeper（或者 Kafka 新版嘅 KRaft）負責儲存 broker、partition、consumer group 嘅元數據。我建議你了解埋 KRaft，因為新版 Kafka 已經開始淘汰 ZooKeeper。

我教你消息語義

At-most-once、At-least-once、Exactly-once

我要同你講清楚一件事：消息可能會重複、可能會丟失。你做系統設計嗰陣，一定要根據業務場景揀啱嘅語義。我幫你拆解三種：At-most-once（最多一次，可能丟）、At-least-once（最少一次，可能重複）、Exactly-once（恰好一次，最難實現）。

我舉個例子你就明：收錢嘅場景，你一定想要 Exactly-once，但實現起嚟好複雜。所以我教你一個實戰技巧——用 At-least-once 加上冪等性（idempotent）。即係話，就算重複收到同一消息，都唔會重複扣錢。呢個做法係業界最常見嘅。

Topic + Partition

Topic 係邏輯分類（例如 order-events）。Partition 係物理分片，你要記住：加 Partition 可以增加並行度，呢個就係水平擴展嘅核心。

Consumer Group

我要你理解呢個概念：同一 Group 嘅 Consumer 瓜分 Partition，每條消息只會被一個 Consumer 處理。但唔同 Group 可以各自消費同一 Topic，呢個就係 pub/sub 嘅精髓。

At-least / Exactly-once

At-least-once：consumer commit offset 喺處理之後，所以可能重複。Exactly-once：Kafka 0.11+ 支援事務，但你要配合冪等 producer 先至做到。

Event Sourcing + CQRS

我建議你了解呢個進階模式：將「事件」當做 source of truth，寫入 Kafka。CQRS 就係讀寫分離——寫入 event store，讀取用 materialized view。

Topic + Partition 擴展

我提醒你：Partition 數量決定並行度。太多 partition 會增加 overhead，太少會成為瓶頸。我建議你按吞吐量嚟估算。

Consumer Group

每個 Group 獨立消費，呢個好適合「一個事件要觸發多個下游」嘅場景。例如 Notification Service 同 Analytics Service 各自一個 Group，互不影響。

消息語義

你要記住呢條規則：金融場景一定要 Exactly-once 或冪等。日誌、metrics 用 At-least-once 就夠，重複唔緊要。

Event Sourcing

我建議你用事件流做 audit trail、replay、debugging。配合 CQRS 讀寫分離，查詢可以用專用嘅讀模型，效能會好好多。

Kafka vs RabbitMQ — 我嘅建議

我成日俾人問呢個問題，我直接講：Kafka 適合高吞吐、持久化、event streaming 同日誌聚合；RabbitMQ 適合傳統 MQ、routing 靈活嘅場景，例如 task queue。你揀邊個要睇場景——高吞吐 event 就用 Kafka，複雜 routing 就用 RabbitMQ。
