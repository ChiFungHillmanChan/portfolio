Coding Agent 設計

拆解 AI 寫 code 嘅核心流程——Plan、Execute、Verify 無限循環

Plan 階段 — 準備工作

點樣壓縮 context、搵返相關 code，先至開始真正寫 code

我要同你講 Coding Agent 嘅核心設計。成個流程係一個 loop：Plan（計劃）→ Execute（執行）→ Verify（驗證）。如果驗證通過，就返俾用戶；如果唔 pass，就再 loop 一次。Plan 階段係第一步，目的係準備好所有背景資料，令 Agent 有足夠 context 去理解應該寫啲乜嘢 code。

Plan 階段嘅四個技巧

  1. 1 Compacting（壓縮）——用另一個 LLM 將長篇嘅 context 縮短成摘要，節省 token。我發現好多系統會用平啲嘅 model 做呢步，專門做 summarization。呢個係最基本嘅優化手法。
  2. 2 Shell Search（Terminal 搜尋）——直接用 terminal 指令（例如 ls、grep）搵檔案同搜尋內容。呢個方法快、簡單，適合做基礎嘅文字搜尋。建議一定要配合埋。
  3. 3 Semantic Search（語義搜尋）——用 vector database 做語義相似度搜尋，搵返真正相關嘅 code，而唔係淨係靠關鍵字。呢個係比較進階嘅做法，但效果好好。
  4. 4 Sub-Agents（子 Agent）——將某啲專門任務交俾另一個 Agent 處理，佢有自己嘅 context window，唔會影響主 Agent。我覺得呢個係最靈活嘅方法，適合複雜任務。

核心目標

Plan 階段嘅目標係：準備好足夠嘅 context，令 Agent 有能力寫出正確嘅 code。唔係盲目開始寫，而係先了解背景、搵返相關檔案、壓縮唔重要嘅內容。呢個就係 Plan 嘅精髓。

Execute 階段 — 真正寫 Code

有咗 context 之後，點樣實際執行、調用工具、生成代碼

準備好 context 之後，就到 Execute 階段。呢個階段係真正寫 code、調用工具、執行任務嘅地方。我會教你三個主要嘅執行方式：MCP（外部工具）、Scripts（本地腳本）、Codegen（Agent 自己生成代碼）。呢三個方法各有用途，要視乎情況選擇。

Execute 階段嘅三個執行方式

  1. 1 MCP（Model Context Protocol）——用 MCP 連接外部工具，例如 database query、API 調用、檔案系統操作。我認為呢個係標準化嘅工具接口，好適合連接各種外部服務。
  2. 2 Scripts（腳本）——直接執行本地嘅測試腳本。例如 run unit tests、linter、formatter 等。呢個方法快速、直接，適合本地開發環境。建議用嚟做快速驗證。
  3. 3 Codegen（代碼生成）——Agent 自己寫 code，可能會用 sub-agent 嚟處理呢個任務，保留主 Agent 嘅 context window。我發現呢個係最靈活嘅方法，適合複雜嘅代碼生成需求。

MCP 嘅優勢

標準化接口，容易整合各種外部工具。一次設定，反覆使用。好適合需要連接多個外部服務嘅情況。

Scripts 嘅優勢

快速、直接，唔使額外設定。適合本地測試同驗證。可以直接調用現有嘅開發工具，例如 pytest、eslint 等。

Codegen 嘅優勢

最靈活，可以生成任何類型嘅代碼。用 sub-agent 處理可以保留主 Agent 嘅 context，避免污染。適合複雜任務。

Execute 階段嘅核心

Execute 階段係實際執行任務嘅地方。三個方法各有優勢：MCP 適合外部工具、Scripts 適合本地快速驗證、Codegen 最靈活。我建議視乎任務需求選擇合適嘅方式，甚至可以組合使用。

Verify 階段 — 驗證成果

執行完之後，點樣驗證代碼係咪真係啱？

執行完任務之後，就到 Verify 階段。呢個階段係驗證 Agent 寫出嚟嘅 code 係咪正確、有冇 bug、符唔符合要求。我會教你三個主要嘅驗證方法：Unit Tests（單元測試）、LLM-as-Judge（用 LLM 做評審）、Visual Testing（視覺測試）。如果驗證唔通過，就 loop 返去 Plan 階段再嚟過。

Verify 階段嘅三個驗證方法

  1. 1 Unit Tests（單元測試）——跑測試，睇 pass 定 fail。呢個係最客觀、最可靠嘅驗證方法。我建議一定要有 unit tests，唔好淨係靠人手檢查。
  2. 2 LLM-as-Judge（用 LLM 評審）——有啲嘢唔係黑白分明，例如 code style、可讀性、註釋質量。呢啲灰色地帶可以用另一個 LLM 評審。我覺得呢個方法好適合處理主觀判斷。
  3. 3 Visual Testing（視覺測試）——用 Playwright 等工具，透過 MCP server 連接，檢查 UI 輸出係咪正確。我認為呢個係驗證前端 code 嘅必備方法。

完整流程循環

如果 Verify 階段發現問題（測試 fail、LLM 評審唔通過、視覺輸出錯誤），就會 loop 返去 Plan 階段。Agent 會重新準備 context，可能會加入錯誤訊息作為新嘅背景資料，然後再執行、再驗證。呢個 loop 會不斷重複，直到驗證通過為止。我見過有啲系統會設上限，例如最多 loop 3 次，避免無限循環。

Unit Tests 最可靠

有測試就跑測試，pass / fail 好清晰，冇灰色地帶。建議投資時間寫好 test suite，咁 Agent 先可以自動驗證。

LLM-as-Judge 處理灰色地帶

有啲判斷好主觀，例如「code 易唔易讀」、「註釋寫得好唔好」。呢啲可以用 LLM 評審，但要記住：呢個唔係 100% 準確。

Visual Testing 檢查 UI

前端 code 一定要有視覺測試。用 Playwright 等工具，透過 MCP 連接，自動檢查 UI 輸出。呢個係驗證前端必備。

Verify 階段嘅取捨

驗證越嚴格，Agent 產出嘅質量就越高，但同時會慢好多（因為要 loop 多次）。我建議根據項目性質調整：對於關鍵系統，嚴格驗證；對於快速 prototype，寬鬆啲都可以接受。

完整 Loop 總結

Plan（準備 context）→ Execute（實際執行）→ Verify（驗證成果）。如果驗證 pass，返俾用戶；如果唔 pass，loop 返去 Plan 階段再嚟過。呢個就係 Coding Agent 嘅核心設計。三個階段缺一不可，組合埋就係一個完整嘅 AI 寫 code 系統。
