Server vs Serverless

傳統 Server 定 Serverless？兩種完全唔同嘅遊戲規則

Server 方案：完全掌控

你話事，你負責，乜都喺你手

我先同你講傳統 Server 嘅玩法。Server 就係你自己租或者買一台機——可以係雲端嘅 EC2、可以係自己屋企嘅實體機——然後你話事晒所有嘢。

你揀 GPU、揀 RAM、揀 儲存空間。你決定裝咩作業系統、裝咩軟件、開咩 port。你負責做 OS update、security patch、防火牆設定。你話要幾多台就幾多台——但你要自己管理 load balancing 同 auto scaling。

重點特性

完全控制權

你話點改就點改。想裝咩軟件、想用咩 port、想點配 network 都得。冇人阻止你。

Always-on 成本

Server 係 24 小時開住嘅——就算冇人用，你都要畀錢。雲端按小時計，自己嘅機就係電費。

自己管理維護

OS update、security patch、監控、備份——全部你負責。有問題半夜三更都要你起身搞。

手動擴展

流量多咗要自己加 Server。可以寫 auto scaling script，但都係你自己搞，唔會自動幫你。

優點

  • 完全控制硬件同軟件
  • 可以優化到極致
  • 長期穩定流量成本可預測
  • 冇 vendor lock-in

缺點

  • Always-on = 冇流量都照收錢
  • 要有人手管理維護
  • 擴展要時間（手動加機）
  • 你負責所有安全問題

適合場景

流量穩定、有專人維護、長期運行嘅服務。例如公司內部系統、穩定嘅 API、大型網站後端。如果你有技術團隊去管理，Server 長遠嚟講可能平過 Serverless。

Serverless 方案：雲端幫你搞

你只管寫 code，其他嘢雲端供應商幫你搞晒

我再同你講 Serverless 嘅玩法。Serverless 唔係話冇 Server——而係話你唔使管 Server。硬件、作業系統、擴展、維護——全部雲端供應商幫你搞，你只需要 deploy code。

最常見嘅係 AWS Lambda、Google Cloud Functions、Azure Functions。你寫一個 function，上傳，然後每次有 request 就自動幫你執行。用完即棄，冇 request 就唔收錢。

重點特性

雲端揀硬件

你唔使揀 GPU、RAM、Storage。雲端供應商會自動配置適合嘅硬件，你唔需要知道底層跑緊咩機。

自動擴展

流量多咗會自動加 instance，流量少咗會自動減。完全唔使你操心，全自動處理。

按 request 收費

冇 request 就唔收錢——真正嘅 pay-as-you-go。但流量大嘅時候帳單會好驚人。

零維護負擔

OS update、security patch、監控——全部雲端供應商搞。你唔使半夜起身處理服務器問題。

優點

  • 冇流量就免費（真正按用量收費）
  • 自動 scaling，唔使人手操作
  • 零維護，唔使管 Server
  • 快速部署，幾分鐘就上線

缺點

  • 大流量時成本爆升
  • Vendor lock-in（好難搬走）
  • Cold start 延遲問題
  • 你改唔到底層配置

適合場景

流量唔穩定、spiky 嘅服務。例如每日淨係繁忙時間先有流量嘅 App、週末先多人用嘅服務、event-driven 嘅工作（處理上傳、發送通知）。如果你唔想請人管 Server、預算充裕，Serverless 係好選擇。

Cold Start 係咩？第一次 request 嚟嘅時候，Serverless function 要「開機」——要 load code、建立連接。呢個過程可能要幾百 ms 甚至幾秒。如果你需要低延遲，就要考慮呢個問題。

點樣揀：三個關鍵問題

根據流量模式、預算、團隊規模嚟決定

我總結咗三個關鍵因素畀你。你問自己呢三條問題，基本上就知道應該揀 Server 定 Serverless。

  • 1
    流量模式
    如果流量穩定（例如 24 小時都有人用），Server always-on 嘅成本反而平過 Serverless 按 request 收費。但如果流量 spiky（例如淨係繁忙時間先有），Serverless 就抵好多——大部分時間免費。
  • 2
    預算同團隊
    有技術人手去管 Server？預算有限？咁用 Server，長遠平啲。如果冇人手、唔想煩、預算充裕，就用 Serverless——用錢買方便。
  • 3
    控制權 vs 便利性
    需要完全控制硬件同軟件（例如特殊嘅 GPU、自訂網絡設定）？就要用 Server。如果你只係想跑 code、唔想理底層，Serverless 係完美選擇。

我嘅建議

如果你係 startup，一開始用 Serverless——快速上線、唔使管 Server、成本低（因為一開始流量少）。但當你 scale 到一定規模、流量穩定咗，就要考慮搬去 Server——因為大流量下 Serverless 會好貴。

如果你有技術團隊、流量穩定、長期運行，一開始就用 Server——控制權高、成本可預測。但要有人手去管理維護。

混合用都得！好多公司其實係混合用嘅——主要 App 跑喺 Server 度（穩定流量），但簡單嘅 background job（例如發 email、處理圖片）就用 Serverless。唔使二揀一，按需求混搭先係最聰明。
