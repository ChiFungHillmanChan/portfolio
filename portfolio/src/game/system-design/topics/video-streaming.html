Video Streaming 影片串流系統

我教你點樣設計影片編碼、Adaptive Bitrate 同 CDN 分發架構

我同你拆解影片串流系統

上傳、轉碼、分發、播放一條龍

我要你理解一個關鍵概念：用戶上傳一條影片，你唔可以直接俾人睇。你要先做 Transcoding（轉碼）——將佢轉成多種解像度同碼率，然後放上 Object Storage，經 CDN 分發到全球各地。播放器用 HLS 或 DASH 協議，根據用戶網速自動揀唔同清晰度，呢個就係 Adaptive Bitrate。

我教你記住成個流程：佢係一個 DAG Pipeline——上傳 → 轉碼（多解像度）→ 儲存 → CDN 快取 → 播放。每一步都有依賴關係，你設計嘅時候要諗清楚。

我帶你行一次成個流程

  1. 1用戶上傳影片，你要將佢送入 Transcoding Pipeline。我推薦你用 FFmpeg，係業界標準。
  2. 2轉碼成多種解像度（1080p、720p、480p、360p），輸出 HLS 格式（.m3u8 + .ts）或 DASH 格式。你要記住，每個 quality level 都係獨立嘅 playlist。
  3. 3分片存入 Object Storage，同時 Video Metadata DB 記錄影片資訊同 playlist URL。我建議你將 metadata 同實際檔案分開存。
  4. 4CDN 快取熱門影片，用戶播放時從最近嘅 edge node 攞。呢個大幅減少延遲，你一定要善用 CDN。
  5. 5播放器根據網速揀唔同清晰度——呢個就係 Adaptive Bitrate Streaming。我要你明白，呢個機制令到用戶體驗好好多。

我教你 Transcoding 流程

點樣將一條片變成多種清晰度

我要同你解釋清楚：原始影片可能係 4K、單一碼率，你唔可以直接咁送出去。你要用 FFmpeg（或類似工具）生成多條不同解像度嘅串流。HLS 用 .m3u8 做 playlist，列出每個 segment（.ts 檔）嘅 URL。DASH 就用 MPD（XML）做類似嘅嘢。

播放器會根據當前 bandwidth 揀最合適嘅 quality level，動態切換。我要你記住，呢個就係 Adaptive Bitrate Streaming 嘅核心原理——唔使用戶手動揀畫質，系統自動幫佢搞掂。

Transcoding 多解像度

我建議你將一條源片至少轉成 1080p、720p、480p、360p。每種有獨立 playlist，player 按網速自動揀。低端裝置都睇到，高速網絡有靚畫質。

HLS / DASH 協議

你要知道兩種主流協議：HLS（Apple 主導）用 .m3u8 + .ts；DASH（開放標準）用 MPD + 分片。兩者都 support adaptive bitrate。我建議你優先用 HLS，兼容性最好。

CDN Edge Caching

熱門影片嘅 .m3u8 同 .ts segment 會 cache 喺 CDN edge 節點。全球用戶就近下載，大幅減輕 origin server 壓力。我要你諗清楚 cache invalidation 策略。

DAG Pipeline

成個流程係：上傳 → 轉碼（可以並行多個 quality）→ 儲存 → 通知。我建議你用 DAG 管理依賴關係，某個 step 失敗可以單獨 retry，唔使重頭嚟。

實戰要點

我要你設計影片串流系統時注意呢幾點

轉碼時間

我同你講，一條 1 小時嘅片轉碼可能要幾十分鐘。你要用 worker 集群並行處理，或者 pre-transcode 熱門片。用戶唔會等你慢慢轉。

Segment 長度

HLS segment 一般 2 到 10 秒。我建議你用 6 秒——太短 overhead 多，太長切換 quality 嘅反應就唔夠快。呢個係個平衡點。

進度通知

轉碼完成後你要通知前端。我建議你用 Webhook 或者 Server-Sent Events，唔好用 polling——浪費資源。更新影片狀態為「可播放」。

存儲成本

你要留意，多解像度 = 多倍存儲成本。我嘅建議係：熱門片保留所有 quality，冷門片只留 360p 或者用分層存儲（hot/cold tier），慳錢之餘唔影響用戶體驗。

我嘅總結

影片串流系統嘅核心就係三個字：Transcoding + CDN + Adaptive Bitrate。你掌握呢三個概念，就可以設計出一個 production-ready 嘅串流架構。記住，先從小規模開始，但架構要可以水平擴展。
