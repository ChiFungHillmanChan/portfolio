Metrics & Logging 監控日誌系統

Metrics & Logging 係咩？

我教你點樣收集系統運行數據，做監控、告警同排查問題

我問你：你嘅系統出咗事，你點樣知？CPU 爆咗、請求慢、錯誤率升……你靠嘅就係 Metrics（指標） 同 Logs（日誌） 嚟掌握系統狀態。我要你分清楚：Metrics 係數字（CPU%、QPS、延遲），Logs 係文字記錄（邊個 API 幾時出錯）。

我教你成條 pipeline 嘅流程：呢啲數據由 App Server 產生，經過 Log Agent 收集，再送去 Kafka 做 buffer，最後存入 Time Series DB 或 Elasticsearch，用 Grafana 畫圖表畀你睇。你記住呢個流程。

我教你成個流程點樣運作

  1. 1App Server 產生 Log（寫入檔）同 Metrics（記憶體計數）。Log Agent（Fluentd、Filebeat）會定期讀取或接收推送。我要你記住：呢個係數據嘅源頭。
  2. 2Log Agent 將數據 push 去 Kafka。我同你講 Kafka 嘅角色好重要——佢做 buffer：流量突增時下游唔使即時處理，可以慢慢消費，唔會丟數據。
  3. 3時序數據（CPU、QPS）去 Time Series DB；Log 文字去 Elasticsearch。Grafana 畫圖表同設定告警規則。我建議你記住呢個分工。

收集模式：Push vs Pull

我教你兩種唔同嘅方式去收集 Metrics 同 Logs

我要你分清楚兩種模式。Push 模式：App 主動將數據送去 collector。適合分散式系統，每個服務自己 push。缺點係 collector 要處理大量連接。

Pull 模式：Collector（例如 Prometheus）主動去 scrape 每個 target 嘅 metrics endpoint。集中管理，容易加減 target。Prometheus 就用呢種。我建議你兩種都要識。

Push vs Pull 收集

Push：App → Collector。Pull：Collector 定期去 App 攞。我教你個原則：Log 通常 Push，Metrics 兩種都得。你要根據場景揀。

Kafka Buffer

我要你記住：高流量時下游可能處理唔切。Kafka 做 buffer，解耦 producer 同 consumer，唔會丟數據。呢個係關鍵設計。

Time Series DB

InfluxDB、Prometheus、VictoriaMetrics 專為時序數據設計，壓縮好、查詢快。我建議你一定要學識用至少一種。

Alert Rules

我教你設定告警：CPU > 80% 就發 Slack、錯誤率升就 call on-call。Grafana Alertmanager 係最常用嘅組合。

實戰要點

我教你監控系統設計嘅關鍵決策

Push vs Pull 收集

我建議你：Push 適合分散式、短生命週期服務（例如 Kubernetes Pod）。Pull 適合集中管理、穩定嘅 target list。你要根據你嘅架構去揀。

Kafka Buffer

我要你記住一個原則：千萬唔好 App 直接寫 DB。Kafka 做 buffer，下游可以慢慢處理，流量高峰都頂得住。呢個係我嘅實戰經驗。

Time Series DB

我教你點揀：Metrics 用 InfluxDB、Prometheus；Log 用 Elasticsearch。唔好用 MySQL 存時序數據，會好慢。你要記住呢個 trade-off。

Alert Rules

我建議你學識 Golden Signals：延遲、吞吐量、錯誤率、飽和度。設定合理 threshold，避免告警疲勞。呢四個指標係你嘅起步點。

我推薦嘅常見組合

Prometheus + Grafana 做 metrics；Elasticsearch + Kibana 做 log 搜尋；Loki 係 lightweight 嘅 log 方案。Kafka 串起成條 pipeline，確保高可用。你跟住呢個組合去學就啱。
